# diffo 需求设计文档

## 1. 文档信息

- 文档类型：需求设计文档
- 产品形态：终端运行的交互式 Git Review CLI/TUI
- 目标读者：产品设计、架构设计、开发实现、测试、后续 Agent 接入
- 文档目标：明确产品范围、核心交互、功能需求、数据需求、非功能需求与验收标准

## 2. 产品概述

diffo 是一款运行于终端中的 Git Diff Review 工具，面向开发者在本地仓库内对变更进行浏览、审阅、标记与评论。工具默认聚焦于当前工作区变更，即未暂存（unstaged）与已暂存（staged）内容；同时兼容 `git diff` 语法，可对 commit、branch、range 等对象进行 review。

产品提供类代码审查场景的终端体验：

- 中央主区域展示 Git Diff
- 右侧边栏展示变更文件目录
- 支持 inline / split diff 视图切换
- 支持 diff 语义高亮与代码语法高亮
- 支持文件级已审阅标记
- 支持单行与多行评论
- 支持通过 CLI 和 Agent 读取评论与审阅状态

产品不以替代 Git 平台 PR Review 为目标，而是聚焦本地 review、提交前自检、Agent 协作与轻量审阅记录沉淀。

## 3. 设计目标

### 3.1 核心目标

1. 让开发者在终端内高效完成本地变更审阅。
2. 支持“先 review 再 commit / push”的本地工作流。
3. 支持人与 Agent 共享评论与 review 状态。
4. 在保留 Git 原生语义的前提下，提供稳定、一致的 TUI 交互。
5. 提供可扩展主题系统，兼容 Base16/Base24 主题规范，并默认采用 Catppuccin 主题。

### 3.2 非目标

1. 不实现完整 Git 客户端能力，如 commit、rebase、merge 冲突解决。
2. 不直接替代 GitHub/GitLab 的远程 PR 审查 API。
3. 第一阶段不实现多人实时协作同步。
4. 第一阶段不实现复杂代码语义分析，仅围绕 diff 与评论信息组织。

## 4. 目标用户与使用场景

### 4.1 目标用户

- 本地开发者：提交代码前做自查与标记
- Code Reviewer：在本地查看某个分支、commit 或 diff range
- AI Agent 使用者：让 Agent 读取本地 review 评论并辅助修复
- 终端重度用户：希望避免频繁切换到 GUI 平台

### 4.2 典型场景

1. 开发者修改若干文件后，用工具查看 unstaged + staged diff，并逐文件标记已审阅。
2. 开发者对某段实现写下评论，随后调用 Agent 读取评论并给出修复建议。
3. Reviewer 打开 `main...feature` 的 diff，在终端中审阅变更并记录意见。
4. 某文件被再次修改后，原有“已审阅”状态自动失效，避免误判为已检查。

## 5. 关键术语

- Working Tree Review：针对当前工作区变更的 review，默认包含 unstaged + staged 变更。
- Review Target：本次 review 的 Git 比较目标，可为默认工作区、某个 commit、commit range、branch range 等。
- File Review State：文件级审阅状态，包含 `reviewed` 与 `unreviewed`。
- Comment Thread：挂载于某文件某一行或连续多行上的评论记录。
- Anchor：评论或审阅状态绑定的定位信息，用于标识文件、行范围与对应 diff 上下文。
- Snapshot：某次 review target 解析后生成的稳定视图数据。

## 6. 产品范围

### 6.1 第一阶段范围

- 打开默认工作区 diff
- 支持传入 `git diff` 风格目标进行 review
- 文件列表浏览与文件切换
- inline / split 视图切换
- 文件标记已审阅 / 未审阅
- 文件变更后自动回退为未审阅
- 单行评论与多行评论
- 评论持久化
- 通过 CLI 查询评论与 review 状态
- 为 Agent 提供结构化读取输出
- 主题系统兼容 Base16/Base24 主题规范，默认主题为 Catppuccin
- 代码高亮支持常见语言，并能在终端能力不足或文件过大时优雅降级

### 6.2 后续可扩展范围

- 评论状态机：open / resolved
- 全局搜索评论
- 按作者或标签过滤评论
- 与 GitHub/GitLab 评论格式互转
- 键盘宏、自定义快捷键
- blame、上下文文件预览、符号级导航

## 7. 用户流程

### 7.1 默认工作流

1. 用户在仓库目录执行 CLI。
2. 系统解析当前 Git 仓库与默认 review target。
3. 主界面展示 diff，右侧展示文件树。
4. 用户浏览文件、切换 diff 视图、移动光标。
5. 用户可对文件标记“已审阅”。
6. 用户可对单行或选中多行添加评论。
7. 系统将 review 状态与评论写入本地存储。
8. 后续再次打开时，系统恢复状态；若文件内容发生变化，则自动重置已审阅状态。

### 7.2 Agent 协作流程

1. 用户在 diff 上写评论。
2. Agent 通过 CLI 子命令读取评论及其文件、行范围、上下文信息。
3. Agent 基于评论内容执行解释、修复或生成补丁。
4. 用户再次打开 CLI 确认处理结果。

## 8. 功能需求

## 8.1 Review Target 解析

### 8.1.1 默认行为

- 当用户未传入目标时，默认 review 当前工作区中的 unstaged + staged 变更。
- 默认视图应尽可能与以下组合语义一致：
  - 未暂存：`git diff`
  - 已暂存：`git diff --cached`
  - UI 中以统一文件树呈现，但保留状态来源信息供内部判断

### 8.1.2 显式目标

工具应支持传入与 `git diff` 兼容的比较目标，包括但不限于：

- `commit`
- `commit^!`
- `A..B`
- `A...B`
- `branch`
- `branch1..branch2`
- `--cached`
- `-- pathspec`

### 8.1.3 目标解析原则

- 尽量复用 Git 原生命令与语义，不自行重新定义 diff 规则。
- 无法解析的目标应返回明确错误信息。
- 对于 pathspec 过滤，文件树与评论查询结果均应按过滤后的视图工作。

## 8.2 主界面

### 8.2.1 布局

默认布局为三部分：

- 顶部状态栏：仓库、分支、review target、主题、统计信息
- 中央主视图：diff 内容区
- 右侧边栏：文件树与文件 review 状态

可选区域：

- 底部状态栏：快捷键提示、当前模式、评论提示
- 弹层/面板：评论输入框、帮助面板、主题切换面板

### 8.2.2 文件树

文件树至少展示：

- 文件相对路径
- Git 状态摘要（新增、修改、删除、重命名等）
- 是否已审阅
- 评论数量

文件树支持：

- 上下移动
- 快速定位当前 diff 文件
- 跳转到下一个未审阅文件
- 按状态筛选：全部 / 未审阅 / 有评论

## 8.3 Diff 浏览

### 8.3.1 视图模式

需支持两种模式：

- Inline Diff
- Split Diff

要求：

- 提供快捷键在两种模式间切换
- 切换后尽量保持当前文件与光标位置语义连续
- 两种模式下都可创建单行/多行评论

### 8.3.2 浏览能力

支持以下能力：

- 文件内滚动
- hunk 间跳转
- 文件间跳转
- 跳转到首个未审阅文件
- 折叠/展开文件 diff（可选，建议纳入第一阶段）

## 8.4 代码高亮

### 8.4.1 高亮类型

系统应支持两层高亮：

- Diff 语义高亮：新增行、删除行、上下文行、hunk header、文件 header、冲突或异常状态
- 代码语法高亮：基于 Tree-sitter grammar 与 highlight query 对关键字、字符串、注释、类型、函数、数字、操作符等 token 着色

### 8.4.2 语言识别

语言识别应按以下优先级执行：

1. 文件扩展名
2. 文件名特例，如 `Makefile`、`Dockerfile`、`build.zig`
3. shebang
4. Git 属性或后续配置项

无法识别语言时，仅保留 diff 语义高亮。

### 8.4.3 Tree-sitter 要求

- V1 采用 Tree-sitter 作为语法高亮与 AST 基础能力
- 每种受支持语言需声明对应 grammar、highlight query 与维护状态
- 当 grammar 或 query 缺失时，自动降级为纯 diff 语义高亮
- AST 能力在 V1 主要服务语法高亮，后续可用于符号导航、函数级折叠、结构化评论锚点重定位
- 对 diff 片段无法稳定解析时，应优先读取完整 old/new 文件内容进行解析，再映射回 diff 行

### 8.4.4 高亮范围

- inline diff 中，新增行和删除行都应保留代码语法高亮
- split diff 中，左右两侧分别基于 old/new 内容进行语法高亮
- 语法高亮不应破坏行号、评论标记、选区和 reviewed 状态标记
- 二进制文件、超大单行、不可解码文本应跳过语法高亮并给出降级展示

### 8.4.5 配置需求

应支持以下配置：

- 启用/禁用语法高亮
- 启用/禁用 Tree-sitter AST 解析
- 配置启用的 Tree-sitter grammar 列表
- 指定主题中的代码 token 色槽
- 限制最大高亮文件大小
- 指定某些文件类型跳过高亮

### 8.4.6 性能要求

- 语法高亮不得阻塞常规滚动体验
- 大文件可采用懒加载、按视口高亮或缓存策略
- Tree-sitter 解析失败、超时或 grammar 不可用时必须降级，不影响 diff 浏览
- 当语法高亮失败时，界面仍应正常展示 diff 内容

## 8.5 文件审阅状态

### 8.5.1 状态定义

文件支持以下状态：

- `unreviewed`
- `reviewed`

### 8.5.2 标记行为

- 用户可在当前文件上通过快捷键切换为 `reviewed`
- 用户也可手动重置为 `unreviewed`

### 8.5.3 自动失效规则

以下任一条件发生时，文件的 `reviewed` 状态自动回退为 `unreviewed`：

- 文件 diff 内容发生变化
- 文件对应的 Git blob / patch 指纹发生变化
- review target 改变导致文件比较结果变化

### 8.5.4 保留策略

- 评论默认保留，不因文件重新修改而自动删除
- 但评论可能进入“锚点失配”状态，系统需尽量重定位，并在无法重定位时标记异常

## 8.6 评论系统

### 8.6.1 评论类型

支持两类评论：

- 单行评论
- 多行评论

### 8.6.2 评论能力

每条评论至少包含：

- 唯一 ID
- 文件路径
- 锚点信息
- 行范围
- 评论正文
- 创建时间
- 作者标识
- review target 标识

### 8.6.3 评论创建

- 用户可对当前光标行添加评论
- 用户可进入选择模式，选定连续多行后创建评论
- 评论应附带足够上下文，以便文件变动后尝试重定位

### 8.6.4 评论展示

- 在 diff 行旁显示评论标识
- 在文件树中显示评论计数
- 支持打开当前文件评论列表
- 支持从评论跳转回对应 diff 位置

### 8.6.5 评论存储原则

- 评论记录按文件组织与索引
- 同时保留跨文件全局索引，便于 CLI 查询与 Agent 读取

### 8.6.6 评论读取

CLI 必须支持：

- 读取全部评论
- 按文件读取评论
- 读取结构化 JSON 输出
- 输出行号/多行范围/锚点状态

### 8.6.7 Agent 兼容要求

Agent 读取评论时必须可获得：

- 文件路径
- 行号或行范围
- 评论正文
- 锚点上下文
- 当前是否匹配最新 diff
- 当前 review target 标识

## 8.7 CLI 接口需求

### 8.7.1 交互式启动

示例：

```bash
diffo
diffo --cached
diffo HEAD~1..HEAD
diffo main...feature -- src/
```

### 8.7.2 非交互查询

示例能力：

```bash
diffo comments list
diffo comments list --file src/main.zig
diffo comments list --json
diffo review status
diffo review status --json
```

### 8.7.3 脚本友好输出

- 支持 `--json`
- 返回稳定字段名
- 非 0 退出码体现错误状态

## 8.8 配置与主题

### 8.8.1 默认主题

- 默认采用 Catppuccin 配色主题

### 8.8.2 主题能力

- 支持 Base16/Base24 主题规范的导入、映射与切换
- 支持通过配置文件指定默认主题
- 支持运行时切换主题

### 8.8.3 配置项

至少支持：

- 默认主题
- 默认 diff 模式
- 是否启用代码高亮
- 代码高亮最大文件大小
- 文件树位置与宽度
- 评论作者名
- 是否显示行号
- 是否默认折叠已审阅文件

## 8.9 数据持久化

系统至少需持久化以下数据：

- 仓库级配置
- review session 元数据
- 文件已审阅状态
- 评论数据
- 主题偏好

要求：

- 不污染 Git 仓库工作内容
- 默认按 XDG Base Directory Specification 写入用户级 config/state/data/cache 目录
- 提供清晰的数据迁移策略

## 9. 快捷键需求

建议默认快捷键如下：

- `j` / `k`：上下移动
- `h` / `l`：左右移动或面板切换
- `tab`：主视图与文件树切换
- `]c` / `[c`：下一个 / 上一个 hunk
- `]f` / `[f`：下一个 / 上一个文件
- `v`：切换 inline / split
- `r`：切换当前文件 reviewed 状态
- `c`：当前行添加评论
- `V`：进入多行选择模式
- `enter`：确认或跳转
- `?`：帮助
- `q`：退出

最终快捷键应支持后续自定义，但第一阶段可先固定。

## 10. 非功能需求

## 10.1 性能

- 中小型仓库（< 500 变更文件）启动时间应可接受
- 常规滚动与跳转应保持流畅，无明显卡顿
- 评论读取命令在正常数据量下应快速返回

## 10.2 兼容性

- 支持 Linux/macOS，Windows 通过常见终端环境运行
- 支持常见 ANSI 终端能力
- 对非真彩终端应提供降级显示策略

## 10.3 可用性

- 键盘优先
- 错误提示明确
- 帮助文档与快捷键面板可即时查看

## 10.4 可维护性

- review target 解析、diff 渲染、评论存储应解耦
- 主题系统需可插拔
- CLI 查询接口字段稳定，便于 Agent 长期依赖

## 10.5 可靠性

- 任意持久化失败不得损坏原仓库数据
- 存储写入采用原子写策略
- 数据结构需包含版本字段，支持未来升级

## 11. 数据需求

## 11.1 文件审阅状态数据

至少包含：

- 仓库标识
- review target 标识
- 文件路径
- 文件 patch/blob 指纹
- reviewed 状态
- 更新时间

## 11.2 评论数据

至少包含：

- comment_id
- repository_id
- review_target_id
- file_path
- anchor 类型
- start_line
- end_line
- old/new side 信息
- diff 上下文片段
- body
- author
- created_at
- updated_at
- anchor_match_status

## 12. 错误场景

系统应明确处理以下场景：

- 当前目录不是 Git 仓库
- Git 不可用或版本不兼容
- review target 非法
- 文件已删除或重命名导致评论定位失败
- 存储文件损坏
- 终端尺寸过小无法完整渲染

## 13. 安全与边界

- 工具只读访问 Git 内容，除评论与状态持久化外不修改源码
- 不自动执行用户仓库中的脚本
- 非交互查询默认不输出敏感环境信息

## 14. 验收标准

### 14.1 核心验收

1. 在 Git 仓库根目录执行 `diffo`，可正常展示 unstaged + staged diff。
2. 可通过参数打开 commit / branch / range diff。
3. 主界面存在 diff 主区和右侧文件树。
4. 可通过快捷键切换 inline / split 视图。
5. 可将文件标记为 reviewed，并在重启后恢复。
6. 文件变更后，原 reviewed 状态自动失效。
7. 可添加单行评论和多行评论。
8. CLI 可按文件和 JSON 格式输出评论。
9. Agent 读取到的评论结果中包含文件、行范围和锚点信息。
10. 默认主题为 Catppuccin，且主题系统可加载并映射 Base16/Base24 主题。
11. 常见代码文件在 inline / split diff 中具备语法高亮，关闭高亮后仍可正常浏览。

### 14.2 质量验收

1. 持久化失败不会造成仓库内容损坏。
2. 错误信息可定位问题原因。
3. 在典型中等规模仓库中界面交互流畅。

- resolved 评论状态

### V1.2

- 更丰富的 Agent 工作流集成
